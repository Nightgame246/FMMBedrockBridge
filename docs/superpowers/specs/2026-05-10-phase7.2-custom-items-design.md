# Design: Phase 7.2 — Custom Items für Bedrock (Split)

**Datum:** 2026-05-10
**Scope:** EliteMobs Custom Items (2D-Texturen) auf Bedrock-Clients sichtbar machen — GUI-Icons und gehaltene Items

---

## Ausgangslage

EliteMobs-Items (Waffen, Rüstung, Shop-Icons) nutzen Javas `custom_model_data`-System mit custom PNG-Texturen im EM Resource Pack. Geyser übersetzt `custom_model_data` nicht automatisch — Bedrock-Clients sehen das Basis-Item (z.B. `diamond_sword`) ohne Custom-Textur.

**Referenz:** Nitrosetups auf dem Velocity-Proxy macht das identisch — eigene `.mcpack` mit Item-Texturen + Registrierung in Geyser via `GeyserDefineCustomItemsEvent`. Das resultierende GUI (Custom Icons in Inventar-Fenstern) sieht auf Bedrock korrekt aus.

**Bekannt unbekannt (wird in 7.2a geklärt):** Existieren 3D-Item-Modelle (Blockbench-Waffen) in EliteMobs? Das bestimmt ob Phase 7.2c nötig ist.

## Phase 7.2b — 2D Custom Item Bridge

### Mechanismus

Geysers `GeyserDefineCustomItemsEvent` feuert beim Geyser-Start. Dort registrieren wir jedes EliteMobs-Item als Bedrock Custom Item. Die Texturen werden in unser bestehendes Bedrock-Pack eingebettet.

### Datenfluss

```
Spigot-Plugin Start:
  EliteMobsItemScanner.scan()
    → liest EM Resource Pack
    → extrahiert List<EMCustomItem>
    → schreibt em-items.json in plugins/FMMBedrockBridge/output/ (dasselbe Verzeichnis wie fmm-entities.json)

Geyser-Extension Start:
  BedrockResourcePackGenerator
    → liest em-items.json
    → kopiert PNG-Texturen nach textures/items/em/
    → bettet in unser Bedrock-Pack ein

  GeyserDefineCustomItemsEvent
    → für jedes EMCustomItem: event.register(CustomItemData)

  GeyserDefineResourcePacksEvent
    → unser Pack (mit EM-Texturen) wird an Bedrock-Clients gesendet

Bedrock-Client joined:
    → bekommt Pack
    → Custom Icons in EliteMobs-Shops sichtbar
    → Gehaltene EM-Items haben Custom-Textur
```

### Neue Klassen

**`EliteMobsItemScanner`** — `de.crazypandas.fmmbedrockbridge.bridge`

```
Aufgabe: EM Resource Pack scannen, custom_model_data Mappings extrahieren
Input:   Pfad zum EM Resource Pack (aus config.yml)
Output:  List<EMCustomItem>, serialisiert als em-items.json
```

**`EMCustomItem`** — `de.crazypandas.fmmbedrockbridge.bridge` (Record)

```java
record EMCustomItem(
    String javaMaterial,      // z.B. "diamond_sword"
    int customModelData,      // z.B. 1001
    String texturePath,       // relativ zum EM-Pack, z.B. "assets/elitemobs/textures/item/sword_1.png"
    String bedrockIdentifier  // z.B. "fmmbridge:em_sword_1"
)
```

### Geänderte Klassen

**`BedrockResourcePackGenerator`** (Geyser-Extension)
- Neue Methode: `embedEliteItemTextures(List<EMCustomItem>)` — kopiert PNGs nach `textures/items/em/`
- Neue Methode: `generateItemTextureDefs(List<EMCustomItem>)` — schreibt `textures/item_texture.json` Einträge

**`FMMBridgeExtension`** (Geyser-Extension)
- Neuer Event-Handler: `@Subscribe onDefineCustomItems(GeyserDefineCustomItemsEvent)`
- Liest `em-items.json`, registriert jeden Eintrag via `event.register(CustomItemData)`

### Config

```yaml
# config.yml (Spigot-Plugin)
elite-items:
  enabled: true
  resource-pack-path: "plugins/EliteMobs/resourcepack"  # Pfad zum EM Java Resource Pack — exakter Pfad wird in 7.2a Research ermittelt
```

### Scope-Grenzen

- **IN:** 2D-Items (flache PNG-Textur via `custom_model_data`)
- **OUT:** 3D-Items mit Blockbench-Geometry (→ 7.2c)
- **OUT:** Bedrock Forms / NPC-Dialoge (→ 7.3)
- **OUT:** Animierte Items
- **OUT:** Items anderer Plugins (nur EliteMobs)

### Erfolgskriterien

1. Bedrock-Client öffnet EliteMobs-Shop → Custom Icons sichtbar (nicht Vanilla-Items)
2. Bedrock-Spieler hält EM-Waffe in der Hand → Custom-Textur sichtbar
3. Kein Crash, kein Log-Spam beim Start
4. Java-Clients sind nicht betroffen
