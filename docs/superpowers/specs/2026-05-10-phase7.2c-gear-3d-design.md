# Design: Phase 7.2c — 3D Gear Items für Bedrock

**Datum:** 2026-05-10
**Scope:** EliteMobs 3D-Gear-Items (Schwerter, Äxte, Sensen, Dreizacks) auf Bedrock-Clients sichtbar machen — gehaltene Items zeigen Bedrock-Geometrie statt Vanilla-Item

---

## Ausgangslage

Phase 7.2b hat 2D-UI-Icons registriert (flat PNG via `custom_model_data`). EliteMobs hat zusätzlich 3D-Gear-Items — Waffen deren Java-Modell `elements`-Cubes enthält (Blockbench-Style). Diese werden von Geyser nicht automatisch übersetzt.

**Aus 7.2a Research (TestServer01):**
- 24 registrierte 3D-Held-Weapon-Einträge auf: `diamond_sword` (CMD 1-10), `diamond_axe` (CMD 1-5), `diamond_hoe` (CMD 1-5), `netherite_sword` (CMD 1-3), `wooden_sword` (CMD 1)
- Modell-Pfade unter `elitemobs:gear/` (z.B. `elitemobs:gear/bronze_sword`)
- Jedes Modell hat `elements`-Array, `textures`-Map (1 Textur-Slot), `display`-Block, kein `parent`

**Out of scope (→ 7.2d oder ignorieren):**
- Helmets (`iron_horse_armor` CMD) — anderer Item-Slot-Typ
- Bögen/Crossbows — haben Pull-Animations-Varianten (CMD pulling_0/1/2)
- Coins, Maps, UI-Icons — kein held-item Kontext
- Items mit `_icon` im Modell-Namen — GUI-only

---

## Architektur

### Bedrock Mechanismus

Bedrock unterstützt **Attachables** — Resource-Pack-Definitionen die sagen: "Wenn der Spieler Item X hält, zeige diese 3D-Geometrie." Zusammen mit Geysers `GeyserDefineCustomItemsEvent` (bekannt aus 7.2b) funktioniert das so:

1. Geyser registriert `fmmbridge:em_bronze_sword` als Custom-Item für `minecraft:diamond_sword` CMD 1
2. Bedrock-Client bekommt das Item → sucht im Resource Pack nach `attachables/fmmbridge_em_bronze_sword.json`
3. Attachable verweist auf `geometry.fmmbridge.em_bronze_sword` → Bedrock rendert die 3D-Geometrie in der Hand
4. Inventar-Slot: `item_texture.json` liefert die Textur-PNG als flaches Icon (akzeptabel für 7.2c)

### Datenfluss

```
Spigot (Startup, elite-items.enabled UND elite-items.gear-3d.enabled):
  EliteMobsItemScanner.scan3DGear():
    → findet alle 3D-Gear-Items (elements vorhanden + Pfad enthält "gear/" + kein "_icon" + nicht iron_horse_armor)
    → schreibt bedrock-pack/em-gear-models/<bedrockKey>.json  (Raw Java-Model-JSON)
    → kopiert Textur-PNG nach bedrock-pack/em-gear-textures/<bedrockKey>.png
    → schreibt bedrock-pack/em-gear-items.json

Geyser-Extension (PreInitialize):
  ResourcePackBuilder.embedGearItems(inputDir):
    → liest em-gear-items.json + em-gear-models/*.json
    → JavaItemGeometryConverter: elements → Bedrock geo.json
    → generiert attachable-Definition pro Item
    → bettet ein:
        geometry/em/<bedrockKey>.geo.json
        attachables/fmmbridge_<bedrockKey>.json
        textures/items/em/gear/<bedrockKey>.png
        textures/item_texture.json (Einträge für 3D-Items ergänzen)

  GeyserDefineCustomItemsEvent:
    → registriert jeden 3D-Gear-Item als CustomItemData (gleich wie 7.2b)
```

## Neue Klassen

### `EMGearItem` — Spigot, `de.crazypandas.fmmbedrockbridge.bridge`

```java
record EMGearItem(
    String javaMaterial,      // z.B. "minecraft:diamond_sword"
    int customModelData,      // z.B. 1
    String bedrockKey,        // z.B. "em_bronze_sword"
    String sourceModelPath,   // absoluter Pfad zur Java-model-JSON
    String sourceTexturePath  // absoluter Pfad zur PNG
)
```

Serialisiert nach `bedrock-pack/em-gear-items.json`.

### `JavaItemGeometryConverter` — Geyser-Extension, `de.crazypandas.fmmbridgeextension`

Liest Java-model-JSON (`elements`, `textures`, `texture_size`) und schreibt Bedrock `.geo.json`.

**Koordinaten-Konvertierung:**
- Java-Item-Space: `from`/`to` in 0-16 Pixel-Einheiten, Ursprung bei 8,8,8
- Bedrock-Geo-Space: Ursprung bei 0,0,0
- Formel: `bedrock_coord = java_coord - 8` (für alle x, y, z)
- Element-Rotation: gleicher `angle`, gleiche `axis`; `origin` ebenfalls `- 8`
- UV: Java `[u1, v1, u2, v2]` → Bedrock `"uv": [u1, v1], "uv_size": [u2-u1, v2-v1]`

**Alle Cubes landen in einem einzigen `root`-Bone** (Java-Items haben keine Bone-Hierarchie — `groups` ist nur Blockbench-Organisationshilfe).

**Generiertes geo.json Format:**
```json
{
  "format_version": "1.12.0",
  "minecraft:geometry": [{
    "description": {
      "identifier": "geometry.fmmbridge.em_bronze_sword",
      "texture_width": 64,
      "texture_height": 64,
      "visible_bounds_width": 4,
      "visible_bounds_height": 4,
      "visible_bounds_offset": [0, 2, 0]
    },
    "bones": [{
      "name": "root",
      "pivot": [0, 0, 0],
      "cubes": [ /* konvertierte elements */ ]
    }]
  }]
}
```

`texture_width`/`texture_height` aus dem `texture_size`-Array im Java-Model (Default: [64, 64]).

### Geänderte Klassen

**`EliteMobsItemScanner`** (Spigot):
- Neue Methode `scan3DGear()`  — selber Sub-Pack-Walk wie `scan()`, anderer Filter:
  - `elements` vorhanden
  - Modell-Pfad enthält `"gear/"`
  - Modell-Name enthält NICHT `"_icon"`
  - Base-Material ist NICHT `iron_horse_armor`
- Exportiert Raw Java-model-JSON + PNG, schreibt `em-gear-items.json`

**`FMMBedrockBridge.onEnable()`** (Spigot):
- Zusätzlicher Block wenn `elite-items.gear-3d.enabled`: ruft `scan3DGear()` auf, schreibt Outputs via `writeEmGearItemsJson()`

**`ResourcePackBuilder`** (Geyser-Extension):
- Neue Methode `embedGearItems(Path inputDir)`: liest em-gear-items.json, konvertiert via `JavaItemGeometryConverter`, schreibt geo.json + attachable + Textur + item_texture.json-Einträge

**`FMMBridgeExtension`** (Geyser-Extension):
- `onPreInitialize`: ruft `embedGearItems(inputDir)` nach `embedEliteItems(inputDir)`
- `onDefineCustomItems`: zweiter Loop über em-gear-items.json, registriert `CustomItemData` pro Gear-Item

## Bedrock Attachable Format

Pro Item eine Datei unter `attachables/fmmbridge_<bedrockKey>.json`:

```json
{
  "format_version": "1.10.0",
  "minecraft:attachable": {
    "description": {
      "identifier": "fmmbridge:em_bronze_sword",
      "materials": { "default": "entity_alphatest" },
      "textures": { "default": "textures/items/em/gear/em_bronze_sword" },
      "geometry": { "default": "geometry.fmmbridge.em_bronze_sword" },
      "render_controllers": [ "controller.render.item_default" ]
    }
  }
}
```

## Scope-Grenzen

- **IN:** 3D-Gear auf `diamond_sword`, `diamond_axe`, `diamond_hoe`, `netherite_sword`, `wooden_sword`
- **OUT:** Helmets (`iron_horse_armor`) → 7.2d
- **OUT:** Bögen / Crossbows (Pull-Animations) → 7.2d
- **OUT:** Coins, Maps, UI-Icons → ignorieren oder 7.2d
- **OUT:** `display`-Block-Übersetzung (Positioning-Hints) → Phase 8
- **OUT:** 3D-Inventar-Icon (gerendertes Preview) → Phase 8

## Erfolgskriterien

1. Bedrock-Spieler hält EM-Schwert (`diamond_sword` CMD 1) → sieht `bronze_sword` 3D-Geometrie statt Vanilla-Diamond-Sword
2. Inventar zeigt Textur-PNG als flaches Icon (akzeptiert)
3. Kein Crash, kein Log-Spam beim Proxy-Start
4. Java-Clients sind nicht betroffen
5. Config-Toggle `gear-3d.enabled: false` deaktiviert sauber (kein geo.json / Attachable im Pack)
