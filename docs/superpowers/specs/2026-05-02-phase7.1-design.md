## Ausgangslage

Bedrock-Spieler sehen FMM-Custom-Models seit Phase 1–6.4 korrekt inkl. Animationen, Static-Entities und Head/Body-Orientation. Drei UI-Probleme bleiben:

1. **BossBar zeigt Vanilla-Mob-Typ** (z.B. "Evoker | 2") statt EliteMobs-Custom-Name
2. **Nametag folgt dem Mob nicht beim Bewegen** — bleibt an alter Position hängen
3. **Healthbar fehlt** — Java zeigt 3-zeiligen Nametag (HP-Zahl / farbiger Balken / Mob-Name), Bedrock nur Name

Phase 7 wird in drei Sub-Phasen aufgeteilt:

- **7.1 (dieses Design):** BossBar + Nametag-Lag + Healthbar (Quick Wins, niedriges Risiko)
- **7.2 (später):** GUI-Icons / Custom Items für Bedrock
- **7.3 (später):** Java-Popup-Menüs → Bedrock Forms (Cumulus API)

## Architektur-Entscheidungen

### Healthbar selbst bauen (nicht von EliteMobs übernehmen)

Wir bauen die Healthbar als String aus Unicode-Block-Zeichen mit §-Farbcodes selbst, gespeist aus `realEntity.getHealth() / getMaxHealth()`.

**Begründung:**
- EliteMobs hat keine garantierte Public-API für seine Nametag-Komponenten
- Reflection auf EM-Internals würde uns an EM-Versionen koppeln
- Real-Entity HP ist die kanonische Quelle und wird automatisch von Java- und Bedrock-Damage-Events mutiert → Java-↔-Bedrock-Sync ist trivial
- Phase 7.2/7.3 werden ohnehin EM-spezifisch — Phase 7.1 bleibt bewusst entkoppelt

### Passive Sync über Tick-Polling (kein Damage-Listener)

Wir registrieren keinen `EntityDamageEvent`-Listener. Stattdessen liest `FMMEntityData.tick()` jeden Server-Tick die HP und vergleicht den erzeugten Nametag-String mit dem zuletzt gesendeten — Metadata-Paket nur bei Änderung.

**Begründung:**
- Tick-Loop existiert bereits (für Position/Animation-Sync)
- Kein zusätzlicher Event-Bus-Overhead
- Nicht-zerbrechlich gegenüber HP-Änderungen aus untypischen Quellen (Heilung, Wither-Effekt etc.)

### EliteMobs als softdepend, nicht hard-depend

`plugin.yml`: EliteMobs zu `softdepend` hinzufügen — wenn nicht installiert, BossBar-Resolver passiv, Plugin lädt sauber.

### NametagBuilder — Spezifikation

Eingabe: `realEntity` (`org.bukkit.entity.LivingEntity`), `modeledEntity` (FMM `ModeledEntity`)
Ausgabe: `Component` mit 3 Zeilen, getrennt durch `\n`

```
Zeile 1: HP-Zahl  ─ "12.34 / 19.89"  in weiß
Zeile 2: Healthbar ─ "▌▌▌▌▌▌▌▌▌▌"   farbig nach HP-Anteil
Zeile 3: Mob-Name  ─ realEntity.customName() oder FMM displayName
```

**Healthbar-Logik:**
- Länge: 10 Zeichen (final, kein Config in Phase 7.1)
- Anteil = `health / maxHealth`
- Aktuelle-HP-Zeichen in Farbe: ≥66% grün (§a), 33–66% gelb (§e), <33% rot (§c)
- Verbrauchte HP in dunkelgrau (§8)
- Konkrete Glyphen (gefüllt vs. leer) werden beim Implementieren visuell verifiziert — Spec legt HP-Anteil-Mapping und Farb-Tiers fest, exakte Block-Zeichen sind Implementierungs-Detail
- Bei `maxHealth == 0` oder `health < 0`: Healthbar-Zeile weglassen, nur HP-Zahl + Name senden
- Bei Static-Entity (kein realEntity-HP-Zugriff): Healthbar weglassen

**Style-Konstante:** Glyphen + Farbcodes als Konstanten in `NametagBuilder` — kein Config in Phase 7.1, kann in Phase 8 (Polish) konfigurierbar werden.

### BossBarTitleResolver — Spezifikation

Eingabe: BossBar-UUID (aus BOSS_EVENT-Paket)
Ausgabe: ersetzter Title (`Component`) oder `null` (Paket unverändert durchlassen)

**Mapping-Mechanismus:** in der Implementierungs-Plan-Phase final entschieden. Zwei Kandidaten:
- **Bevorzugt:** EliteMobs API exposed `EliteEntity.getBossBar()` o.ä. → beim Boss-Spawn Mapping registrieren
- **Fallback:** beim ersten BOSS_EVENT-Paket einer unbekannten UUID nahe FMM-Entities einen "best match" wählen

Spec legt fest: Resolver muss UUID → Component-Title liefern. Wie er das tut, wird in der Plan-Phase nach EliteMobs-API-Recherche entschieden.

## Datenfluss

### Fluss 1: BossBar-Title-Replacement

```
EliteMobs Boss spawnt
  → Vanilla-BossBar erstellt (UUID X, Title "Evoker | 2")
  → Server sendet BOSS_EVENT (action=ADD oder UPDATE_NAME) Paket
  → PacketInterceptor.onPacketSend hört BOSS_EVENT ab
    → BossBarTitleResolver.resolve(uuid X)
       → ja, FMM-tracked → Real-Entity-customName als neuen Title
       → nein → null, Paket unverändert
  → Bedrock-Client rendert BossBar mit korrektem Namen
```

Java-Spieler bekommen den Vanilla-Title nicht — wir filtern nur auf Bedrock-Spieler-Pakete (über `event.getPlayer()` Check), weil Java weiterhin den von EM gesetzten Custom-Title via Vanilla-Mechanismus sieht.

### Fluss 2: Nametag-Sync (continuous)

```
FMMEntityData.tick() (jeden Tick = 20 Hz)
  → syncPosition() (existiert)
  → syncAnimation() (existiert)
  → syncNametag() (NEU)
    → currentName = NametagBuilder.build(realEntity, modeledEntity)
    → if (!currentName.equals(lastSentName)):
       → für jeden Bedrock-Viewer: packetEntity.sendNameMetadata(currentName, true, viewer)
       → lastSentName = currentName
    → sonst: kein Paket
```

`sendNameToViewer()` (existiert) wird beim Spawn weiterhin den initialen Nametag senden. Die `tick()`-basierte Sync übernimmt nachfolgende Updates.

### Fluss 3: Position-Lag-Diagnose

Position-Sync läuft bereits jeden Tick via `PacketEntity.teleport(realLoc, viewers)`. Lag-Ursachen werden in der Plan-Phase identifiziert. Vermutete Ursachen:

- `PacketEntity.teleport()` sendet RELATIVE_MOVE bei kleinen Bewegungen → Bedrock interpoliert nicht sauber
- Bedrock erwartet vollständige TELEPORT-Pakete für Nametag-tragende Entities

Spec legt fest: **Position-Updates an Bedrock-Viewer müssen so gesendet werden, dass der Nametag mit dem Mob mitläuft.** Konkrete Implementierung (Schwellenwert anpassen / immer TELEPORT senden / etc.) wird in der Plan-Phase nach Code-Analyse von `PacketEntity.teleport()` festgelegt.

### Fluss 4: Java-↔-Bedrock HP-Sync

Damage von Java- oder Bedrock-Spieler mutiert die selbe Real-Entity. Beide Seiten sehen denselben HP-Stand:

```
Damage (Java oder Bedrock)
  → Real-Entity HP sinkt
  → FMMEntityData.tick() (nächster Tick) sieht neuen HP
  → NametagBuilder erzeugt neuen String mit aktualisiertem Healthbar
  → wenn anders als lastSentName: sendNameMetadata an Bedrock-Viewer
  → Java-Spieler sehen es ohnehin direkt über Vanilla
```

Kein eigener Damage-Listener nötig. Bedrock-Damage funktioniert über bestehenden Interact-Redirect in `PacketInterceptor`.

## Error Handling

| Situation | Verhalten |
|-----------|-----------|
| EliteMobs nicht installiert | BossBar-Resolver passiv, alle Pakete unverändert, 1× Info-Log beim Plugin-Start |
| `realEntity.customName()` wirft Exception | Fallback FMM displayName → bedrockEntityId |
| `realEntity.getMaxHealth() == 0` | Healthbar-Zeile weglassen, nur HP-Zahl + Name |
| `realEntity.isDead()` während Tick | Kein Update, Cleanup übernimmt `FMMEntityData.destroy()` |
| BossBar-UUID unbekannt | Paket unverändert durchlassen, kein Log |
| Title-Replacement schlägt fehl | Original durchlassen, einmaliges WARNING-Log pro UUID |
| Bedrock-Viewer offline | `viewer.isOnline()`-Check, skip ohne Fehler |
| EM-API-Reflection schlägt fehl | Resolver fällt auf "kein Replace" zurück, BossBar bleibt Vanilla — kein Crash |

**Prinzip:** alle Fehlerpfade führen zu Vanilla-Verhalten oder Skip, nie zu Crash oder Log-Spam. Phase-6.4-Verhalten darf nicht regressen.
