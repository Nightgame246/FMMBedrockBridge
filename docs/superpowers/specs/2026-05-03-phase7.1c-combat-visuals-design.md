# Design: Phase 7.1c — Combat-triggered Visuals (HP / Bar / BossBar Combat-Toggle)

**Datum:** 2026-05-03
**Scope:** Phase 7.1c — Combat-Trigger für BossBar-Visibility + 2-Zeilen-Erweiterung des Nametags (HP-Zahl + Health-Bar). Multi-BossBar-Rejoin-Issue wird organisch durch die neue Combat-Lifecycle-Architektur gelöst.
**Voraussetzung:** Phase 7.1a (BossBar) + Phase 7.1b (Nametag) abgeschlossen und live verifiziert auf TestServer01.
**Branch:** `phase-7.1c`

---

## Ausgangslage

Nach Phase 7.1a + 7.1b sehen Bedrock-Spieler über jedem EM-Boss:
- ✅ Custom Model + Animationen (Phase 1–6.4)
- ✅ BossBar oben am Bildschirm mit styled Name (Phase 7.1a) — **always-visible** (proximity-based)
- ✅ Nametag über dem Mob mit styled Name (Phase 7.1b) — **always-visible**

Das ist nicht 1:1 zum Java-EM-UX:
- Java zeigt BossBar **nur bei Combat** (EnterCombat-Event-getriggert)
- Java zeigt **HP-Zahl + Health-Bar** als separate FakeText-TextDisplays über dem Mob, ebenfalls **nur bei Combat**
- Bedrock-Spieler haben aktuell BossBar always-on (Crossplay-Info-Vorteil) und KEIN HP-Display über dem Mob

Phase 7.1c bringt Bedrock auf Java-EM-Parität:
- BossBar wird **combat-only** (matched Java-EM-Verhalten)
- Nametag wird bei Combat um **2 Zeilen erweitert** (HP-Zahl + Health-Bar) über der existierenden Name-Zeile
- Zusätzlich: das **Multi-BossBar-Rejoin-Issue** aus Phase 7.1a wird organisch gelöst — Combat-Lifecycle macht den Rejoin-Race-Window obsolet

---

## Architektur-Entscheidungen

### Bukkit-Listener auf EM-Combat-Events, kein Polling

Wir registrieren einen `BedrockCombatTrigger` als Bukkit-`Listener` auf `EliteMobEnterCombatEvent` (Priority `HIGHEST` — vor EM's MONITOR-Listener) und `EliteMobExitCombatEvent`. Combat-State wird in den existing Phase 7.1a/b Controllers gehalten. Listener triggert state-changes; kein Polling, keine eigenen Timer.

**Begründung:**
- EM's eigene `BossHealthDisplay` nutzt das gleiche Event-Pattern — bewiesen funktional
- HIGHEST-Priority garantiert dass unser Code VOR EM läuft → unser BossBar-ADD-Paket geht zuerst raus → First-Match-Heuristik claimt unser, suppressed EM's
- Kein Polling-Overhead

### BossBar combat-only, lazy-add at Combat-Enter

`BedrockBossBarController` bleibt am Boss-Spawn erstellt (existing Phase 7.1a), aber initial `setVisible(false)`. Bedrock-Viewers werden in `viewers` Set getrackt aber NICHT mit `bossBar.addPlayer(p)` registriert solange `isInCombat == false`. Bei Combat-Enter: `setVisible(true)` + `addPlayer(p)` für jeden current Viewer. Bei Combat-Exit: `removeAll()` + `setVisible(false)` + `ownUuids.clear()` (damit beim nächsten Combat first-match neu greift).

**Begründung:**
- Lazy-add löst Multi-BossBar-Rejoin-Issue: bei Rejoin außerhalb Combat sind keine BossBar-Pakete im Spiel → keine Race
- Bei Rejoin während aktiver Combat: `addViewer` (vom ViewerManager) ruft `bossBar.addPlayer(p)` weil `isInCombat=true` → ADD-Paket → first-match findet bestehende own-UUID → pass through. Kein Race mit EM weil EM kein neues Combat-Enter-Event feuert für Rejoin
- `setVisible(false)` initial verhindert dass Phase 7.1a-Verhalten (always-visible) versehentlich wiederhergestellt wird

### Nametag wird kontextsensitiv über `NametagTextBuilder`

`BedrockNametagController` wird refactored: kein `Supplier<Component>` mehr (Phase 7.1b-Pattern), stattdessen baut ein neuer `NametagTextBuilder` den Component pro Tick basierend auf `realEntity`, `modeledEntity`, und einem `boolean inCombat` Flag im Controller.

- `inCombat == false` → 1-Zeile (nur Name) — Phase 7.1b-Verhalten
- `inCombat == true` → 3-Zeilen: HP-Zahl / Health-Bar / Name

**Begründung:**
- Single source of truth für Nametag-Text (vs. Supplier-Pattern hatte versteckte Logik im Aufrufer)
- Builder ist pure utility (statisch), gut testbar
- Combat-State im Controller, nicht in der Supplier-Closure → klarere Architektur

### Plugin-Konfiguration als Notausgang

`config.yml`-Schlüssel `phase71c.combat-enabled: true` (default). Wenn `false`:
- `BedrockCombatTrigger` wird nicht registriert
- `BedrockBossBarController` initial `setVisible(true)` + `isInCombat=true` permanent → fällt zurück auf Phase 7.1a-Verhalten (always-visible)
- `BedrockNametagController` `isInCombat=true` permanent → würde immer 3-Zeilen zeigen → unerwünscht; daher der Toggle nur einsetzen wenn man explizit Phase 7.1a-Verhalten zurückwill und mit always-3-Zeilen-Nametag leben kann. Alternative: separate `phase71c.nametag-combat-only: true` Toggle.

**Wir machen's einfach:** EIN Toggle `phase71c.combat-enabled` (default true). Bei `false`: BossBar always-visible, Nametag bleibt bei 1-Zeile (kein Combat-State-Update). Im Code: BedrockCombatTrigger registriert sich nicht, also setCombatState wird nie aufgerufen, isInCombat bleibt false default → Nametag bleibt 1-Zeile, BossBar bleibt invisible (mit Phase 7.1c-Default `setVisible(false)`).

⚠️ **Das hieße: bei `combat-enabled=false` sehen Bedrock-Spieler GAR KEINE BossBar.** Das ist schlecht.

**Korrigierte Logik:** Bei `combat-enabled=false`:
- BedrockBossBarController-Konstruktor liest config: wenn false → `setVisible(true)` initial (Phase 7.1a-Backup-Verhalten)
- isInCombat-Flag spielt keine Rolle (kein Trigger feuert)
- Nametag bleibt 1-Zeile (kein Trigger)

Der Notausgang wiederherstellt Phase 7.1a-Verhalten für BossBar + Phase 7.1b-Verhalten für Nametag.

## Komponenten

### Neue Klassen

| Klasse | Aufgabe |
|---|---|
| `BedrockCombatTrigger` | Bukkit `Listener`. EM-only — wird nur registriert wenn EM-Plugin geladen UND `phase71c.combat-enabled=true`. `@EventHandler(priority=HIGHEST)` auf `EliteMobEnterCombatEvent`, `@EventHandler(priority=MONITOR)` auf `EliteMobExitCombatEvent`. |
| `NametagTextBuilder` | Pure utility (final class, private constructor). Statische Methode `compose(Entity realEntity, ModeledEntity modeledEntity, boolean inCombat) → Component`. Refines die Phase-7.1-alte `NametagBuilder` aber explizit combat-aware. |

### Modifikationen

**`BedrockBossBarController`:**
- Konstruktor: liest `phase71c.combat-enabled` config. Wenn `true`: `setVisible(false)` initial. Wenn `false`: `setVisible(true)` initial (Phase 7.1a-Backup).
- Neues Feld `boolean isInCombat` (default `false`)
- `addViewer(Player)`:
  - `viewers.add(player)` (immer)
  - Wenn `isInCombat`: `bossBar.addPlayer(player)` (sonst skip — wartet auf nächsten Combat-Enter)
- `removeViewer(Player)` unverändert
- Neue Methode `enterCombat()`: `bossBar.setVisible(true)` + `bossBar.addPlayer(p)` für jeden online Viewer + `isInCombat=true`
- Neue Methode `exitCombat()`: `bossBar.removeAll()` + `bossBar.setVisible(false)` + `isInCombat=false` + `ownUuids.clear()`
- Neuer Getter `isInCombat()` für `/fmmbridge debug`-Output

**`BedrockNametagController`:**
- Konstruktor-Refactor: nimmt jetzt `(Entity realEntity, ModeledEntity modeledEntity, TextDisplay textDisplay, Component initialText)` — kein `Supplier<Component>` mehr
- Neues Feld `boolean isInCombat` (default `false`)
- Neue Methode `setCombatState(boolean)` — setter
- `tickUpdate()`: nutzt `NametagTextBuilder.compose(realEntity, modeledEntity, isInCombat)` statt Supplier
- Neuer Getter `isInCombat()` für `/fmmbridge debug`-Output

**`FMMEntityData.createNametagControllerIfNamed()`:**
- Vereinfachung — kein Supplier-Bauen mehr, nur Refs an Controller weitergeben
- Initial-Text via `NametagTextBuilder.compose(realEntity, modeledEntity, false)` für 1-Zeile

**`FMMBedrockBridge.onEnable()`:**
- Wenn EM verfügbar UND `phase71c.combat-enabled=true`: `getServer().getPluginManager().registerEvents(new BedrockCombatTrigger(bridge), this)` in try/catch
- Bei Failure: WARNING log, Phase 7.1c passiv

**`config.yml`:**
- Neuer Block:
```yaml
# Phase 7.1c — Combat-triggered visuals (HP/Bar/BossBar)
phase71c:
  # When true, BossBar appears only during combat (EnterCombat → ExitCombat).
  # Nametag extends to 3 lines (HP / Health-Bar / Name) during combat.
  # When false, BossBar is always-visible (Phase 7.1a behavior) and nametag stays
  # at 1 line. Use false as a safety fallback if the combat-trigger logic breaks.
  combat-enabled: true
```

**`/fmmbridge debug` (FMMBridgeCommand):**
- Existing BossBar-Sektion erweitert: `inCombat: yes/no` pro Controller
- Existing Nametag-Sektion erweitert: `inCombat: yes/no` pro Controller

### NametagTextBuilder — Spezifikation

```java
public final class NametagTextBuilder {
    public static Component compose(Entity realEntity, ModeledEntity modeledEntity, boolean inCombat) {
        Component nameLine = buildNameLine(realEntity, modeledEntity);
        if (!inCombat || !(realEntity instanceof LivingEntity living)) {
            return nameLine != null ? nameLine : Component.empty();
        }
        Component hpLine = buildHpLine(living);
        Component barLine = buildBarLine(living);
        if (hpLine == null || barLine == null) {
            return nameLine != null ? nameLine : Component.empty();
        }
        Component out = hpLine
                .append(Component.newline())
                .append(barLine);
        if (nameLine != null) {
            out = out.append(Component.newline()).append(nameLine);
        }
        return out;
    }

    // buildNameLine: FMM modeledEntity.getDisplayName() (legacy parsed) primary,
    //                 realEntity.customName() fallback — same as Phase 7.1b logic.
    // buildHpLine:   "12.34 / 19.89" (white, %.2f format), null if maxHealth<=0
    // buildBarLine:  10 chars, '█' filled (green/yellow/red ≥66/≥33/<33),
    //                '░' empty (dark-gray). Never fully empty unless dead.
}
```

**Layout (in-combat, top→bottom):**
```
12.34 / 19.89                ← weiß
██████░░░░                   ← grün/gelb/rot + dunkelgrau
『3』 Wilder Alphawolf       ← styled FMM name
```

**Layout (out-of-combat):**
```
『3』 Wilder Alphawolf
```

## Datenfluss

### Fluss 1 — Boss-Spawn (Combat-only initial state)

```
EM/BetterStructures spawnt FMM-Boss
  → existing FMMEntityTracker erkennt
  → existing FMMEntityData.create():
    1. PacketEntity setup (existing)
    2. createBossBarControllerIfElite():
       → bossBar = createBossBar(title, GREEN, SEGMENTED_10)
       → bossBar.setProgress(1.0)
       → bossBar.setVisible(combatEnabled ? false : true)  // <- gated by config
       → isInCombat = false
    3. createNametagControllerIfNamed():
       → world.spawn TextDisplay (existing)
       → controller initial isInCombat=false → tickUpdate returnt 1-Zeile (Name only)
```

### Fluss 2 — Bedrock-Viewer kommt in Range (außerhalb Combat)

```
ViewerManager → FMMEntityData.addViewer(player)
  → existing PacketEntity-Spawn + GeyserUtils-setup
  → bossBarController.addViewer(player):
     1. viewers.add(player)              // merken
     2. wenn isInCombat: bossBar.addPlayer(player)
        sonst: nichts (BossBar bleibt invisible für jetzt)
  → nametagController.addViewer-Pfad: keine Änderung — TextDisplay sichtbar (Phase 7.1b)
```

### Fluss 3 — Combat-Enter

```
Bedrock-Spieler attackiert Boss → Vanilla Damage-Event
  → EliteMobs verarbeitet → feuert EliteMobEnterCombatEvent

  Listener-Reihenfolge (Bukkit-garantiert):
  ├─ BedrockCombatTrigger (HIGHEST) — UNSER Listener feuert ZUERST
  │  1. uuid = event.getEliteMobEntity().getLivingEntity().getUniqueId()
  │  2. bossBarController = bridge.getActiveControllers().get(uuid)
  │     bossBarController.enterCombat():
  │        - bossBar.setVisible(true)
  │        - für jeden viewer: bossBar.addPlayer(viewer) → ADD-Paket
  │        - isInCombat = true
  │     ADD-Paket geht raus → PacketInterceptor.handleBossEvent:
  │        Title-Match findet UNSER Controller, kein own-UUID → claim als own (first match)
  │  3. nametagController = bridge.getActiveNametags().get(uuid)
  │     nametagController.setCombatState(true)
  │     nächster tickUpdate: NametagTextBuilder.compose(...) returnt 3-Zeilen
  │
  └─ EliteMobs's BossHealthDisplay (MONITOR) — feuert DANACH
     1. EM erstellt eigene BossBar mit gleichem Title
     2. EM's bossBar.addPlayer(targetPlayer) → ADD-Paket
     3. PacketInterceptor.handleBossEvent: Controller hat schon own-UUID → captured EM's UUID + suppressed
```

### Fluss 4 — Tick-Loop während Combat

```
existing BedrockEntityBridge.tickAll() (alle 2 Ticks)
  → für jede FMMEntityData: data.tick()
    → existing syncPosition + animation
    → bossBarController.tickUpdate():  (existing Phase 7.1a)
       - progress = realEntity.getHealth() / getMaxHealth()
       - color tier + setProgress/setColor wenn geändert
       - Pakete mit own-UUID → pass-through
    → nametagController.tickUpdate():  (modified)
       - newComponent = NametagTextBuilder.compose(realEntity, modeledEntity, true)
       - 3-Zeilen Component: HP-Line + Bar-Line + Name-Line
       - wenn != lastText: textDisplay.text(component) (Java-suppressed via PacketInterceptor)
```

### Fluss 5 — Combat-Exit

```
EM Combat-Timeout (~10 Sek inactivity) → feuert EliteMobExitCombatEvent
  → BedrockCombatTrigger.onExitCombat:
    1. uuid lookup
    2. bossBarController.exitCombat():
       - bossBar.removeAll()       → REMOVE-Paket pro viewer
       - bossBar.setVisible(false)
       - isInCombat = false
       - ownUuids.clear()           // beim nächsten Combat first-match neu greifen
    3. nametagController.setCombatState(false)
       - nächster tickUpdate: 3-Zeilen → 1-Zeile
```

### Fluss 6 — Boss-Death (mid-combat)

```
existing FMMEntityData.destroy():
  → existing PacketEntity, BossBarController.cleanup, NametagController.cleanup, activeControllers/Nametags.remove
  → BedrockCombatTrigger weiß nichts vom Death — kein eigener Cleanup nötig (Lookups returnen null danach)
```

### Fluss 7 — Bedrock-Spieler Rejoin während Boss in Combat

```
Spieler disconnects mid-combat
  → ViewerManager.onPlayerLeave → data.removeViewer(player)
  → bossBarController.removeViewer:
    - viewers.remove(player)
    - bossBar.removePlayer(player) (REMOVE-Paket gesendet, weil isInCombat=true)

Player reconnects
  → ViewerManager.setOnPlayerReady → data.addViewer(player)
  → bossBarController.addViewer:
    - viewers.add(player)
    - WENN isInCombat (Combat noch aktiv server-side): bossBar.addPlayer(player) → ADD-Paket
    - Title-Match: Controller hat schon own-UUID (vom EnterCombat-Event vorher) → pass-through
    - EM feuert KEIN neues Combat-Enter-Event (Spieler ist server-side-state-mäßig schon im Combat)
    → KEIN Doppel-BossBar-Race ✓
```

**Multi-BossBar-Rejoin-Issue: ORGANISCH GELÖST** durch Combat-only Lifecycle.

## Error Handling

| Situation | Verhalten |
|---|---|
| EM nicht installiert | `BedrockCombatTrigger` wird nicht registriert. BossBars bleiben gated by `combat-enabled` config: bei `true` invisible (kein Trigger), bei `false` always-visible (Phase 7.1a). Nametag-controller `isInCombat=false` permanent → 1-Zeile. Akzeptabel weil ohne EM auch keine FMM-EM-Bosse spawnen. |
| EM Combat-Event-API ändert sich | try/catch um `registerEvents`. Bei Failure: WARNING, Phase 7.1c passiv. |
| Combat-Enter-Event für non-FMM-Mob | `bridge.getActiveControllers().get(uuid)` returnt null → Listener no-op. |
| Combat-Enter mit `getLivingEntity() == null` | Null-check vorne, return. |
| Combat-Exit für unbekannten Boss (z.B. Boss schon despawnt) | Map-Lookup null → no-op. |
| Mehrere Bedrock-Spieler im Combat-Range | `enterCombat()` iteriert `viewers` Set, addPlayer für alle. (Java-Parität "per-attacker" hier nicht 1:1, akzeptiert.) |
| Bedrock-Spieler verlässt Range mid-Combat | `removeViewer` → `bossBar.removePlayer`. `isInCombat` bleibt true (Boss noch in Combat). Andere Viewer sehen weiter. |
| Boss stirbt mid-Combat | `FMMEntityData.destroy()` cleant alles. EM feuert kein ExitCombat mehr (Entity weg) — kein Trigger nötig. |
| `realEntity.getHealth() > getMaxHealth()` | Bar-fraction clamped auf [0,1]. HP-Zahl zeigt raw Wert (selbsterklärend). |
| `getMaxHealth() == 0` | NametagTextBuilder skip HP+Bar lines, returnt nur Name-Line. |
| **EM's eigene HP/Bar TextDisplays erscheinen auf Bedrock parallel** ⚠️ | Risk-Check in Task 8 (Manual Test). Falls Doppel-HP-Display: Follow-up Fix (TextDisplay-Suppress oder Title-Match). |
| First-Match-Heuristik schlägt fehl bei Combat-Enter | Bedrock sieht 2 BossBars beim Combat. Existing Logging im PacketInterceptor markiert das. Notausgang: `combat-enabled=false`. |
| PacketEvents-Listener-Reihenfolge | Bukkit-Event-Priority (HIGHEST→MONITOR) garantiert Combat-Event-Reihenfolge. PacketEvents sieht Pakete in derselben Reihenfolge. ✓ |

### Prinzipien
1. Phase 7.1a + 7.1b dürfen nicht regressen — Phase 7.1c ist purer Visibility-Toggle auf existing Architektur
2. Fail-graceful — wenn EM-API ändert / Listener kaputt: Phase 7.1c passiv, Rest läuft
3. Config-Notausgang `phase71c.combat-enabled: false` stellt Phase 7.1a-Backup-Verhalten her

- BossBar erscheint nur bei Combat-Enter, verschwindet bei Combat-Exit

- Multi-BossBar-Rejoin-Issue (Phase 7.1a) ist gelöst — keine doppelten BossBars mehr nach Disconnect/Reconnect
