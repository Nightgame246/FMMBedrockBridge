# Design: Phase 7.1b — Bedrock Nametag (always-visible Name)

**Datum:** 2026-05-03
**Scope:** Phase 7.1b — Custom-Name als always-visible Nametag über FMM-Mobs auf Bedrock-Clients. HP-Zahl + Health-Bar als Combat-trigger werden in Phase 7.1c separat gespec't.
**Voraussetzung:** Phase 7.1a (BossBar) abgeschlossen und live auf TestServer01 verifiziert.
**Branch-Stand:** `main` mit allen Phase-7.1a-Commits.

---

## Ausgangslage

Bedrock-Spieler sehen seit den Phase 1–6.4 + 7.1a Implementierungen:
- ✅ Custom-Modelle für FMM-Mobs (Phase 1–6.4)
- ✅ Korrekte Body/Head-Orientation (Phase 6.4)
- ✅ BossBar oben am Bildschirm mit styled Name für EliteMobs-Bosse (Phase 7.1a)

Was ihnen fehlt:
- ❌ **Nametag über dem Mob** — Java-Spieler sehen den Custom-Name (z.B. "『3』 Wilder Alphawolf") als schwebenden Text über jedem benannten Mob. Bedrock sieht nichts.

Aus Phase-7.1-Findings (Memory): GeyserUtils Custom-Entities (registriert via `EntityUtils.setCustomEntity`) honorieren KEINE nachträglichen `WrapperPlayServerEntityMetadata`-Custom-Name-Updates. Der Phase-7.1-Versuch über die Fake-PIG-Entity (`sendNameMetadata`) hat deshalb nicht funktioniert. Ein anderer Pfad ist nötig.

---

## Architektur-Entscheidungen

### Eigene `TextDisplay`-Bukkit-Entity, vor Java versteckt

Pro FMM-Mob mit `customName != null` spawnen wir eine echte `org.bukkit.entity.TextDisplay`-Entity (server-side) mit dem Custom-Name als Text. Position wird im bestehenden Tick-Loop auf `realEntity.location + Y-offset` synchronisiert (kein Passenger-Mount, weil das Bedrock's Hidden-Real-Mob-Pfad bricht). Java-Spieler sehen die TextDisplay nicht — `PacketInterceptor` blockiert SPAWN_ENTITY/ENTITY_METADATA/ENTITY_TELEPORT für diese Entity-IDs an Nicht-Floodgate-Spieler. Bedrock-Spieler bekommen die Pakete durchgereicht — Geyser übersetzt TextDisplay zu Bedrock-Nametag-Entity.

**Begründung:**
- TextDisplay → Bedrock-Nametag ist ein **bewiesen funktionierender Geyser-Pfad** (pre-FMM-boneFix sah Fabi genau das — FMMs eigene TextDisplay-Spawns wurden auf Bedrock als Nametags gerendert).
- EM nutzt selbst `FakeText` (TextDisplay-Wrapper) für Mob-Health-Bars in `BossHealthDisplay.java` — gleicher Pattern, gleiche Library.
- Kein Eingriff in die Bedrock-Entity-Definition (`.entity.json`) oder den Custom-Entity-Replace-Pfad → keine neuen Risiken im Bedrock-Resource-Pack.
- Pattern-konsistent mit Phase 7.1a (`BedrockNametagController` parallel zu `BedrockBossBarController`).

### Plugin-agnostic — Trigger ist `customName != null`, nicht "ist EliteMobs-Boss"

Wir hooken nicht in EliteMobs- oder andere Plugin-spezifische APIs ein. Ein Mob bekommt einen Bedrock-Nametag genau dann, wenn `realEntity.customName()` einen Wert liefert. Damit decken wir automatisch ab:
- EliteMobs-Bosse (customName via `EliteEntity.setName`)
- BetterStructures-Mobs (falls customName gesetzt)
- Andere Magma-Plugins (Tower Defense, ResourcePackManager)
- Vanilla-Mobs mit name-tag (die FMM-getrackt sind)

### Java-Suppress via spiegelbildliche Logik im PacketInterceptor

Aktuell hat PacketInterceptor `hiddenEntityIds: Map<Integer, Set<Player>>` — "Entity X ist für Spieler-Set Y versteckt" (per-Spieler, für Real-Mob-Hide-from-Bedrock).

Neu: `javaHiddenEntityIds: Set<Integer>` — "Entity X ist für ALLE Java-Spieler versteckt" (per-Entity-Type). Floodgate-Check entscheidet wer Java ist.

Diese Spiegelbild-Trennung ist sauber: hide-from-Bedrock und hide-from-Java haben getrennte Lifecycle-Sources und müssen unabhängig managed werden.

### Position-Sync via Tick-Loop, nicht Passenger-Mount

Die naheliegende Idee, TextDisplay als Passenger des Real-Mobs zu mounten (Bukkit handled Position-Sync automatisch), funktioniert nicht clean: Bedrock-Spieler sehen den Real-Mob nicht (durch unseren PacketSuppressor versteckt). Geyser kennt das Mount-Verhältnis aber kann den Parent nicht resolved → Position der TextDisplay auf Bedrock unklar.

Alternative: standalone TextDisplay, im existierenden Tick-Loop teleportiert — gleiche Mechanik wie wir für PacketEntity (Fake-PIG) schon nutzen. Bukkit deduplicates ENTITY_TELEPORT-Pakete bei unveränderter Position, also ist der Overhead minimal selbst bei stationären Mobs.

## Komponenten

### Neue Klasse

| Klasse | Aufgabe |
|---|---|
| `BedrockNametagController` | Lifecycle pro FMM-Mob mit Custom-Name. Spawnt + verwaltet eine `TextDisplay`-Bukkit-Entity. Methoden: `tickUpdate()` (Position + Text-Sync, idempotent), `cleanup()` (Entity entfernen + PacketSuppressor cleanup). |

### Modifikationen

| Klasse | Änderung |
|---|---|
| `FMMEntityData` | Feld `@Nullable BedrockNametagController bedrockNametagController`. Im Konstruktor nach BossBar-Setup: `createNametagControllerIfNamed()` — wenn `realEntity.customName() != null`, Controller erzeugen + im Bridge registrieren + im PacketInterceptor `hideFromJava` registrieren. `tick()` ruft `bedrockNametagController.tickUpdate()`. `destroy()` ruft `cleanup()`. |
| `PacketInterceptor` | Neues Feld `Set<Integer> javaHiddenEntityIds = ConcurrentHashMap.newKeySet()` + Methoden `hideFromJava(int entityId)` / `unhideFromJava(int entityId)`. In `onPacketSend`: nach existing hide-from-Bedrock-Check, neuer hide-from-Java-Check für Nicht-Floodgate-Spieler — cancelt SPAWN_ENTITY / ENTITY_METADATA / ENTITY_TELEPORT (auch ENTITY_POSITION etc.) für IDs in `javaHiddenEntityIds`. |
| `BedrockEntityBridge` | Feld `Map<UUID, BedrockNametagController> activeNametags = new ConcurrentHashMap<>()` (key = real-entity UUID) — analog zu `activeControllers`. Cleanup bei `shutdown()`. Public Accessor `getActiveNametags()`. |
| `FMMBridgeCommand` (`/fmmbridge debug`) | Output erweitert um Sektion: Anzahl aktive Nametag-Controller + Text + Bukkit-EntityID pro Controller. |

### Bewusst NICHT in Scope

- **Mehrzeiliger Nametag** (HP/Bar/Name) — Phase 7.1c, combat-triggered.
- **Vehicle/Passenger-Mount** — verworfen, Position-Sync via Tick-Loop ist robust gegen den Hidden-Real-Mob-Pfad.
- **Static-Entity-Nametag** — Static-Entities haben i.d.R. keinen Custom-Name (Möbel ohne Namen). Falls eine StaticEntity doch customName hätte, müsste die Architektur erweitert werden — out-of-scope für 7.1b.
- **Live-Custom-Name-Discovery** für Mobs die ihren Namen NACH dem Spawn bekommen (z.B. EM könnte das mit Delay tun) — Phase 7.1c oder eigener Fix.
- **Animation/Fade-Effekte** — out of scope.

## Datenfluss

### Fluss 1 — Mob-Spawn mit Custom-Name

```
Plugin (EM/BetterStructures/...) spawnt Mob mit setCustomName(...)
  → FMM wrapped als ModeledEntity (existing)
  → FMMEntityTracker.poll() erkennt FMM-Spawn (existing)
  → BedrockEntityBridge.onEntitySpawn() → new FMMEntityData(...)
  → FMMEntityData constructor:
    1. existing PacketEntity setup
    2. existing BossBar setup (Phase 7.1a)
    3. NEU: createNametagControllerIfNamed():
       a) name = realEntity.customName()
       b) wenn name == null → return null (skip, no nametag)
       c) wenn name != null:
          → spawnLoc = realEntity.location + (0, height + 0.3, 0)
          → world.spawn(spawnLoc, TextDisplay.class, td -> {
              td.text(name);
              td.setBillboard(Billboard.CENTER);
              td.setSeeThrough(true);
              td.setDefaultBackground(true);
              hideFromJava(td.getEntityId());  // BEVOR Bukkit das Spawn-Paket sendet
            })
          → controller = new BedrockNametagController(realEntity, spawnedTextDisplay, name)
          → bridge.getActiveNametags().put(realEntity.uuid, controller)
```

**Race-Sicherheit:** `world.spawn(loc, TextDisplay.class, consumer)` ruft den Consumer auf BEVOR das Spawn-Paket an die Welt broadcastet wird (Bukkit-Garantie). `hideFromJava` muss IM Consumer stehen, nicht danach.

### Fluss 2 — Java-Spieler in Range

```
Bukkit broadcast SPAWN_ENTITY für TextDisplay an alle in Range
  → PacketInterceptor.onPacketSend(javaPlayer, packet):
    1. existing hide-from-Bedrock check → kein Match (TextDisplay-ID nicht in hiddenEntityIds)
    2. NEU: hide-from-Java check:
       → wenn !FloodgateApi.isFloodgatePlayer(javaPlayer) UND entityId in javaHiddenEntityIds:
         → event.setCancelled(true), return
  → Java-Client kriegt nichts → kein Doppel-Nametag (Java hat schon Vanilla-Real-Entity-Nametag)
```

### Fluss 3 — Bedrock-Spieler in Range

```
Bukkit broadcast SPAWN_ENTITY für TextDisplay an alle in Range
  → PacketInterceptor.onPacketSend(bedrockPlayer, packet):
    1. existing hide-from-Bedrock check → kein Match
    2. NEU: hide-from-Java check:
       → FloodgateApi.isFloodgatePlayer(bedrockPlayer) == true → kein Cancel
  → Paket geht durch zum Bedrock-Player über Geyser
  → Geyser empfängt SPAWN_ENTITY(TextDisplay) → übersetzt zu Bedrock-Nametag-Entity
  → Bedrock zeigt Text über der TextDisplay-Position
```

### Fluss 4 — Tick-Loop

```
BedrockEntityBridge.tickAll() (existing, alle 2 Ticks)
  → für jede FMMEntityData: data.tick()
    → existing: syncPosition() (PacketEntity)
    → existing: bossBarController.tickUpdate() (Phase 7.1a)
    → NEU: bedrockNametagController.tickUpdate():
       a) wenn realEntity.isDead() oder !isValid(): skip
       b) wenn !textDisplay.isValid(): einmalig WARNING-Log, skip
       c) Position-Sync:
          newLoc = realEntity.location + (0, realEntity.height + 0.3, 0)
          textDisplay.teleport(newLoc)
          → Bukkit dedupliziert (kein Paket bei unveränderter Position)
          → Bei Bewegung: ENTITY_TELEPORT an Viewer, Java cancelt, Bedrock durchgereicht
       d) Text-Sync:
          currentName = realEntity.customName()
          wenn currentName != null UND !currentName.equals(lastText):
            textDisplay.text(currentName)
            → ENTITY_METADATA mit text-Field
            → Java cancelt, Bedrock durchgereicht
          lastText = currentName
```

### Fluss 5 — Mob despawnt / stirbt

```
Mob.remove() / Mob.death()
  → FMMEntityTracker.poll() erkennt fehlende Entity (existing)
  → FMMEntityData.destroy():
    1. existing PacketEntity-Cleanup für alle Bedrock-Viewer
    2. existing BossBar-Cleanup (Phase 7.1a)
    3. NEU: bedrockNametagController.cleanup():
       a) packetInterceptor.unhideFromJava(textDisplay.entityId)
       b) textDisplay.remove() → DESTROY_ENTITIES an alle Viewer
       c) bridge.getActiveNametags().remove(realEntity.uuid)
       → Java: TextDisplay war nie sichtbar → DESTROY ist no-op
       → Bedrock: Nametag verschwindet
```

### Fluss 6 — Plugin Disable

```
FMMBedrockBridge.onDisable() → BedrockEntityBridge.shutdown()
  → für jede FMMEntityData: data.destroy() (cleant alle BossBars + Nametags)
  → activeNametags.clear()
  → packetInterceptor.clear() (cleart auch javaHiddenEntityIds)
```

Nach Server-Stop sind alle TextDisplay-Bukkit-Entities sowieso aus der Welt entfernt. Falls Server während Plugin-Disable noch läuft (Plugin-Reload): cleanup() entfernt sie sauber.

## Error Handling

| Situation | Verhalten |
|---|---|
| `realEntity.customName()` ist null beim Spawn | Kein Controller erzeugt. Plugin-agnostic — wenn niemand einen Namen setzt, kein Nametag. |
| Custom-Name wird mid-life NACHTRÄGLICH gesetzt | Aktuelles Design erkennt das nicht — bekannte Limitierung für 7.1b, ggf. in 7.1c oder Folgephase nachrüsten. |
| Custom-Name wird zur Laufzeit gelöscht (`setCustomName(null)`) | `tickUpdate` setzt Text auf empty Component → leerer Nametag visuell, aber kein "geister-Buchstaben". Controller bleibt aktiv. |
| `realEntity.isDead()` oder `!isValid()` in tickUpdate | Skip teleport + text-update. Cleanup erfolgt durch nächsten FMMEntityTracker-Despawn-Detection-Tick. |
| TextDisplay-Spawn schlägt fehl (Welt nicht geladen, Exception) | try/catch um `world.spawn(...)`. Bei Exception: Controller nicht erstellt, WARNING-Log mit Throwable, Mob bridged ohne Nametag. |
| `textDisplay.isValid() == false` (anderes Plugin removed) | Einmaliges WARNING-Log pro Controller. Kein Re-Spawn (vermeidet Loop). Cleanup beim nächsten Mob-Despawn. |
| Mob teleportiert in andere Welt | TextDisplay versucht cross-world teleport. Bukkit handled das oder schlägt fehl → WARNING + Cleanup. |
| Multi-line customName (`\n`) | TextDisplay rendert Adventure Component mit Newlines korrekt. Akzeptabel. |
| Mehrere FMM-Mobs in Range | Mehrere TextDisplays — Bedrock-Limit unbekannt aber vermutlich unproblematisch (Bedrock zeigt vanilla-Nametags für viele Mobs auch). |
| FMM 2.5.0 sendet eigene TextDisplay-Pakete (neue v26/packets-Subsystem) → Doppel-Nametag-Risiko | **Live-Test verifizieren** in Task 8 der Implementierung. Falls bestätigt: FMM-eigene TextDisplay-Pakete via PacketSuppressor blockieren ODER unsere TextDisplays per Title-Match-Heuristik (analog zu Phase 7.1a BossBar) deduplizieren. |
| PacketEvents-Wrapper API-Break für TELEPORT/METADATA | try/catch. Failure: Pakete laufen durch (Java sieht Doppel-Nametag, Bedrock sieht Nametag — graceful Degradation). |

### Prinzipien
1. Bridge-Kernfunktionen (Phase 6.4 + 7.1a) dürfen nie regressen — alle Nametag-Pfade try/catch-isoliert.
2. Fail-graceful — schlechtester Fall ist "kein Nametag auf Bedrock" (= aktueller Stand vor 7.1b).
3. Java unbeeinflusst — wenn `javaHiddenEntityIds`-Suppress versagt, Java sieht eine extra TextDisplay (sichtbar, hässlich, aber nicht funktional kaputt).

## Erfolgskriterium

Bedrock-Spieler sieht über jedem FMM-Mob mit `customName != null` einen Nametag-Text der dem Java-Vanilla-Nametag entspricht. Nametag folgt der Mob-Position. Java-Spieler sehen kein Doppel-Nametag. Phase 6.4 + 7.1a regressen nicht. Mob ohne Namen → kein Nametag.
